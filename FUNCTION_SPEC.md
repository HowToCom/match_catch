# 기능 명세서

## 1. 문서 개요

### 1.1 문서 목적

본 문서는 분실물 매칭 시스템의 기능 요구사항을 정의하기 위한 문서이다.

백엔드 구현자는 본 문서를 기준으로 회원 관리, 분실물 등록, 습득물 등록, AI 기반 특징 추출, 유사 물건 조회, 매칭 요청, 채팅, 인도 완료, 후기 및 온도 시스템을 구현한다.

본 문서는 단순한 기능 아이디어 정리가 아니라, 실제 구현 과정에서 참조할 수 있는 기능 흐름, 입력값, 출력값, 예외 처리, 상태 관리 기준을 명확히 정의하는 것을 목적으로 한다.

---

### 1.2 문서 적용 범위

본 문서는 다음 기능을 대상으로 한다.

- 회원가입 및 로그인
- 프로필 조회 및 수정
- 사용자 온도 조회
- 활동 내역 조회
- 습득물 등록
- 습득물 이미지 기반 AI 특징 추출
- 분실물 등록
- 유사 습득물 조회
- 매칭 요청
- 매칭 수락 및 거절
- 채팅방 생성
- 메시지 송수신
- 인도 완료 처리
- 후기 작성
- 사용자 온도 변경

단, 관리자 기능, 실시간 알림, 지도 기반 위치 추적, 결제 기능은 본 문서의 기본 구현 범위에서 제외한다.

## 2. 서비스 개요

### 2.1 서비스 목적

본 서비스는 사용자가 분실물과 습득물을 보다 쉽게 연결할 수 있도록 돕는 충남대학교 학생 대상 분실물 매칭 시스템이다.

기존 분실물 게시판은 사용자가 직접 게시글을 검색하고 비교해야 하는 불편함이 있다.  
본 서비스는 습득물 이미지에서 AI가 특징 키워드를 추출하고, 분실자가 입력한 분실물 정보와 비교하여 유사한 습득물을 추천한다.

이를 통해 분실자는 자신의 물건으로 추정되는 습득물을 빠르게 찾을 수 있고, 습득자는 분실자와 안전하게 연락하여 물건을 인도할 수 있다.

---

### 2.2 서비스 핵심 흐름

서비스의 기본 흐름은 다음과 같다.

```text
1. 사용자가 회원가입 및 로그인한다.

2. 습득자는 주운 물건의 사진과 정보를 등록한다.

3. 시스템은 습득물 이미지에서 AI 특징 키워드를 추출한다.

4. 분실자는 잃어버린 물건의 정보를 등록한다.

5. 시스템은 분실물 정보와 등록된 습득물 정보를 비교한다.

6. 분실자는 유사한 습득물 목록을 조회한다.

7. 분실자는 자신의 물건으로 추정되는 습득물에 매칭 요청을 보낸다.

8. 습득자는 매칭 요청을 수락하거나 거절한다.

9. 매칭이 수락되면 분실자와 습득자 간 채팅방이 생성된다.

10. 두 사용자는 채팅을 통해 물건 인도 방법을 조율한다.

11. 물건 전달 후 인도 완료 처리를 한다.

12. 거래 당사자는 후기를 작성할 수 있고, 후기 결과 및 거래 시행에 따라 사용자 온도가 변경된다.

```

---

### 2.3 서비스 주요 특징

| 특징 | 설명 |
|---|---|
| 이미지 기반 등록 | 사용자는 습득물 또는 분실물 등록 시 이미지를 업로드할 수 있다. |
| 학번 기반 회원 관리 | 충남대학교 학생 대상 서비스로, 회원가입 시 이메일 대신 학번을 입력한다. |
| AI 특징 추출 | 업로드된 이미지에서 색상, 형태, 물건 종류 등의 특징 키워드를 자동으로 추출한다. |
| 유사 물건 조회 | 분실물 정보와 습득물의 AI 특징 키워드를 비교하여 유사한 물건을 조회한다. |
| 매칭 요청 시스템 | 분실자는 유사한 습득물에 대해 매칭 요청을 보낼 수 있다. |
| 채팅 기능 | 매칭 요청이 수락된 경우에만 분실자와 습득자 간 채팅방이 생성된다. |
| 상태 기반 관리 | 분실물, 습득물, 매칭은 각각 상태값을 가지며 단계별로 관리된다. |
| 인도 완료 처리 | 물건 전달 후 인도 완료 처리를 통해 거래를 종료할 수 있다. |
| 활동 내역 기록 | 인도 완료 등의 활동 기록이 사용자 프로필에 저장된다. |
| 온도 시스템 | 후기 결과에 따라 사용자 신뢰도 온도가 상승 또는 하락한다. |
| JWT 인증 기반 보안 | 주요 기능은 JWT 인증을 기반으로 접근 권한을 관리한다. |
| 예외 처리 및 권한 검증 | 중복 요청, 권한 없는 접근, 잘못된 상태 변경 등을 서버에서 검증한다. |
| 확장 가능한 구조 | AI 고도화, 알림 기능, 관리자 기능 등을 추가할 수 있도록 구조를 설계한다. |

### 3.3 MVP 구현 범위

초기 MVP에서는 핵심 기능 흐름 구현을 우선 목표로 한다.

#### MVP 핵심 흐름

```text
회원가입
→ 로그인
→ 습득물 등록
→ 이미지 업로드
→ AI 특징 키워드 추출
→ 분실물 등록
→ 유사 습득물 조회
→ 매칭 요청
→ 매칭 수락
→ 채팅방 생성
→ 메시지 송수신
→ 인도 완료
→ 후기 작성
→ 온도 반영
```
#### MVP 핵심 목표

```text
1. 사용자가 분실물 또는 습득물을 등록할 수 있어야 한다.
2. 시스템이 이미지 기반 특징 키워드를 저장할 수 있어야 한다.
3. 분실자가 유사한 습득물을 조회할 수 있어야 한다.
4. 분실자와 습득자가 매칭 요청을 통해 연결될 수 있어야 한다.
5. 매칭 성공 후 채팅을 통해 물건 전달이 가능해야 한다.
6. 거래 완료 후 후기 및 온도 시스템이 동작해야 한다.
```

## 4. 사용자 역할 및 권한 개요

### 4.1 사용자 유형

시스템은 사용자 역할에 따라 접근 가능한 기능과 권한을 구분한다.

---

#### 4.1.1 비회원

비회원은 로그인하지 않은 사용자이다.

비회원은 서비스 이용 전 기본 기능만 사용할 수 있으며,  
회원가입 및 로그인 이후에만 주요 기능 접근이 가능하다.

##### 가능 기능

- 회원가입
- 로그인
- 서비스 소개 확인

##### 제한 사항

- 분실물 등록 불가
- 습득물 등록 불가
- 매칭 요청 불가
- 채팅 기능 사용 불가
- 후기 작성 불가

---

#### 4.1.2 회원

회원은 회원가입 및 로그인을 완료한 사용자이다.

회원은 기본적인 서비스 기능에 접근할 수 있으며,  
분실자 또는 습득자의 역할로 서비스를 이용할 수 있다.

##### 가능 기능

- 프로필 조회
- 프로필 수정
- 활동 내역 조회
- 온도 조회
- 분실물 등록
- 습득물 등록
- 유사 습득물 조회

##### 제한 사항

- 본인이 등록하지 않은 데이터 수정 불가
- 매칭 당사자가 아닌 채팅방 접근 불가
- 인도 완료 전 후기 작성 불가

---

#### 4.1.3 분실자

분실자는 분실물을 등록한 사용자이다.

분실자는 등록한 분실물 정보를 기반으로 유사한 습득물을 조회하고,  
습득자에게 매칭 요청을 보낼 수 있다.

##### 가능 기능

- 분실물 등록
- 분실물 수정
- 유사 습득물 조회
- 매칭 요청
- 매칭 상태 조회
- 채팅 참여
- 인도 완료 처리
- 후기 작성

##### 제한 사항

- 본인이 등록하지 않은 분실물 수정 불가
- 본인이 요청하지 않은 매칭 수정 불가
- 매칭 수락 및 거절 불가
- 인도 완료 전 후기 작성 불가

---

#### 4.1.4 습득자

습득자는 습득물을 등록한 사용자이다.

습득자는 분실자의 매칭 요청을 수락하거나 거절할 수 있으며,  
매칭이 성사된 경우 분실자와 채팅을 진행할 수 있다.

##### 가능 기능

- 습득물 등록
- 습득물 수정
- 매칭 요청 수락
- 매칭 요청 거절
- 채팅 참여
- 인도 완료 처리
- 후기 작성

##### 제한 사항

- 본인이 등록하지 않은 습득물 수정 불가
- 본인이 받은 매칭 요청만 처리 가능
- 인도 완료 전 후기 작성 불가

---

#### 4.1.5 거래 당사자

거래 당사자는 매칭 요청이 수락되어 연결된 분실자와 습득자를 의미한다.

거래 당사자는 채팅 기능을 사용할 수 있으며,  
물건 전달 후 인도 완료 및 후기 작성을 수행할 수 있다.

##### 가능 기능

- 채팅방 접근
- 메시지 송수신
- 인도 완료 처리
- 후기 작성
- 활동 내역 확인

##### 제한 사항

- 매칭과 관계없는 채팅방 접근 불가
- 중복 후기 작성 불가
- 거래 종료 후 상태 변경 제한

## 6. 상태 정의 및 상태 전이

### 6.1 습득물 상태

| 상태 | 설명 |
|---|---|
| REGISTERED | 등록 완료 |
| MATCHING | 매칭 진행 중 |
| DELIVERED | 인도 완료 |

### 6.2 분실물 상태

| 상태 | 설명 |
|---|---|
| REGISTERED | 등록 완료 |
| MATCH_REQUESTED | 매칭 요청 중 |
| MATCHING | 매칭 진행 중 |
| DELIVERED | 인도 완료 |

### 6.3 매칭 상태

| 상태 | 설명 |
|---|---|
| PENDING | 요청 대기 |
| ACCEPTED | 요청 수락 |
| REJECTED | 요청 거절 |
| DELIVERED | 인도 완료 |

### 6.4 상태 전이 규칙

시스템은 분실물, 습득물, 매칭 데이터를 상태 기반으로 관리한다.

각 상태는 정의된 흐름에 따라 변경되며,  
허용되지 않은 상태 변경 요청은 서버에서 차단한다.

---

#### 6.4.1 습득물 상태 전이

습득물은 등록 이후 매칭 진행 및 인도 완료 단계로 상태가 변경된다.

##### 상태 전이 흐름

```text
REGISTERED → MATCHING → DELIVERED
```

##### 상태 전이 규칙

| 현재 상태 | 변경 가능 상태 | 변경 조건 |
|---|---|---|
| REGISTERED | MATCHING | 습득자가 매칭 요청을 수락한 경우 |
| MATCHING | DELIVERED | 거래 당사자가 인도 완료 처리한 경우 |
| DELIVERED | 변경 불가 | 최종 상태 |

##### 제한 사항

- `DELIVERED` 상태의 습득물은 추가 매칭 요청 불가
- `MATCHING` 상태에서는 새로운 매칭 요청 제한 가능
- 정의되지 않은 상태 변경은 허용하지 않음

---

#### 6.4.2 분실물 상태 전이 규칙

분실물은 등록 이후 매칭 요청 및 거래 진행 상황에 따라 상태가 변경된다.

##### 상태 전이 흐름

```text
REGISTERED → MATCH_REQUESTED → MATCHING → DELIVERED
```

##### 상태 전이 규칙

| 현재 상태 | 변경 가능 상태 | 변경 조건 |
|---|---|---|
| REGISTERED | MATCH_REQUESTED | 분실자가 매칭 요청을 보낸 경우 |
| MATCH_REQUESTED | MATCHING | 습득자가 매칭 요청을 수락한 경우 |
| MATCH_REQUESTED | REGISTERED | 습득자가 매칭 요청을 거절한 경우 |
| MATCHING | DELIVERED | 거래 당사자가 인도 완료 처리한 경우 |
| DELIVERED | 변경 불가 | 최종 상태 |

##### 제한 사항

- `DELIVERED` 상태의 분실물은 수정 제한
- 동일 분실물에 대한 중복 매칭 요청 제한 가능
- 정의되지 않은 상태 변경은 허용하지 않음
- 매칭 요청이 거절된 경우 분실물 상태는 `REGISTERED`로 되돌린다.
- 거절 이후 분실자는 다른 습득물에 새로운 매칭 요청을 보낼 수 있다.
- 동일 습득물에 대한 반복 매칭 요청은 제한할 수 있다.
- 하나의 분실물은 동시에 하나의 매칭 요청만 진행할 수 있다.

---

#### 6.4.3 매칭 상태 전이 규칙

매칭은 요청 생성 이후 수락, 거절, 거래 완료 상태로 변경된다.

##### 상태 전이 흐름

```text
PENDING → ACCEPTED → DELIVERED
        └→ REJECTED
```        

##### 상태 전이 규칙

| 현재 상태 | 변경 가능 상태 | 변경 조건 |
|---|---|---|
| PENDING | ACCEPTED | 습득자가 매칭 요청을 수락한 경우 |
| PENDING | REJECTED | 습득자가 매칭 요청을 거절한 경우 |
| ACCEPTED | DELIVERED | 거래 당사자가 인도 완료 처리한 경우 |
| REJECTED | 변경 불가 | 종료 상태 |
| DELIVERED | 변경 불가 | 최종 상태 |

##### 제한 사항

- `REJECTED` 상태의 매칭은 재활성화 불가
- `REJECTED` 상태의 매칭은 다시 `ACCEPTED` 상태로 변경할 수 없다.
- 거절된 매칭에 대한 재시도는 기존 매칭을 재활성화하지 않고 새로운 매칭 요청 생성으로 처리한다.
- `DELIVERED` 상태의 매칭은 수정 불가
- 매칭 수락 후에만 채팅방 생성 가능
- 정의되지 않은 상태 변경은 허용하지 않음

---

#### 6.4.4 상태 변경 공통 규칙

모든 상태 변경은 다음 규칙을 따른다.

1. 상태 변경은 서버에서만 처리한다.
2. 클라이언트가 직접 상태값을 수정할 수 없다.
3. 권한 검증 후 상태 변경을 수행한다.
4. 정의된 상태 전이만 허용한다.
5. 최종 상태(`DELIVERED`)에서는 추가 상태 변경을 제한한다.
6. 상태 변경 시 관련 데이터 상태도 함께 갱신할 수 있다.

---

### 6.5 상태 변경 예외 처리

| 상황 | 처리 방식 |
|---|---|
| 허용되지 않은 상태 변경 요청 | `400 Bad Request` |
| 권한 없는 상태 변경 요청 | `403 Forbidden` |
| 존재하지 않는 데이터 상태 변경 요청 | `404 Not Found` |
| 이미 종료된 상태에 대한 변경 요청 | `409 Conflict` |
| 중복 상태 변경 요청 | `409 Conflict` |

#### 8.9.1 REVIEW-001 후기 작성

##### 기능 설명

거래 당사자가 인도 완료 이후 상대 사용자에 대한 후기를 작성한다.

후기 작성 시 서버는 중복 후기 여부를 검증하고,  
후기 유형에 따라 대상 사용자의 온도를 변경한다.

중복 후기 검증과 온도 변경은 별도 외부 API로 제공하지 않고,  
후기 작성 처리 과정에서 서버 내부 로직으로 수행한다.

---

##### 사용자

- 거래 당사자

---

##### 인증 여부

- 필요

---

##### 요청 데이터

| 필드 | 타입 | 설명 |
|---|---|---|
| match_id | Long | 후기 대상 매칭 ID |
| target_user_id | Long | 후기 대상 사용자 ID |
| review_type | String | 후기 유형 (`POSITIVE`, `NEGATIVE`) |
| content | String | 후기 내용 |

---

##### 처리 로직

1. JWT Access Token을 검증한다.
2. 매칭 존재 여부를 확인한다.
3. 요청 사용자가 해당 매칭의 거래 당사자인지 확인한다.
4. 현재 매칭 상태가 `DELIVERED`인지 확인한다.
5. 후기 대상 사용자가 해당 매칭의 상대 거래 당사자인지 확인한다.
6. 동일 거래에 대해 요청 사용자가 이미 후기를 작성했는지 확인한다.
7. 후기 유형이 `POSITIVE` 또는 `NEGATIVE`인지 검증한다.
8. 후기 정보를 저장한다.
9. 후기 유형에 따라 대상 사용자의 온도를 변경한다.
10. 후기 작성 결과를 반환한다.

## 11. 인증 및 권한 정책

---
### 11.1 인증 방식

본 시스템은 JWT(JSON Web Token) 기반 인증 방식을 사용한다.

사용자는 로그인 성공 이후 Access Token을 발급받으며,  
인증이 필요한 API 요청 시 HTTP Header에 토큰을 포함하여 요청한다.

서버는 요청된 JWT 토큰을 검증한 후 사용자 인증을 수행한다.

---

#### 인증 흐름

```text
로그인 요청
→ 사용자 인증
→ JWT Access Token 발급
→ 클라이언트 저장
→ 인증 필요 API 요청 시 Authorization Header 포함
→ 서버 토큰 검증
→ 요청 처리
```
---

### 11.2 JWT 토큰 사용 방식

JWT Access Token은 로그인 성공 이후 발급한다.

토큰에는 사용자 식별 정보가 포함되며,
서버는 토큰 검증을 통해 요청 사용자를 식별한다.

#### Authorization Header 형식
```http
Authorization: Bearer {ACCESS_TOKEN}
```
---

#### JWT Payload 예시
```JSON
{
  "user_id": 1,
  "student_id": "202512345",
  "username": "user1"
}
```
---

#### JWT 처리 정책

| 항목 | 내용 |
|---|---|
| 인증 방식 | JWT Access Token 기반 인증 사용 |
| 토큰 발급 시점 | 로그인 성공 이후 발급 |
| 토큰 전달 방식 | Authorization Header 사용 |
| Authorization 형식 | `Bearer {ACCESS_TOKEN}` |
| 사용자 식별 방식 | JWT Payload 내부 사용자 정보 사용 |
| 비밀번호 저장 방식 | bcrypt 기반 암호화 저장 |
| 토큰 검증 위치 | 서버 인증 Middleware |
| 인증 실패 처리 | `401 Unauthorized` 반환 |
| 권한 검증 실패 처리 | `403 Forbidden` 반환 |
| 토큰 포함 정보 | `user_id`, `student_id`, `username` |

## 12. 예외 처리 정책

시스템은 클라이언트 요청 처리 과정에서 발생하는 예외 상황에 대해 일관된 응답 형식을 제공한다.

모든 예외 응답은 HTTP Status Code와 JSON 기반 에러 메시지를 포함한다.

---

### 12.1 공통 에러 응답 형식

모든 API의 예외 응답은 다음 JSON 형식을 사용한다.

---

#### 공통 에러 응답 예시

```json
{
  "status": 400,
  "error": "Bad Request",
  "message": "잘못된 요청입니다.",
  "timestamp": "2025-01-01T00:00:00.000Z"
}
```

#### 공통 응답 필드

| 필드 | 타입 | 설명 |
|---|---|---|
| status | Integer | HTTP Status Code |
| error | String | HTTP 상태 이름 |
| message | String | 예외 메시지 |
| timestamp | DateTime | 예외 발생 시간 |


### 12.2 HTTP Status Code 기준

| Status | 사용 상황 |
|---:|---|
| 400 | 잘못된 요청 |
| 401 | 인증 실패 |
| 403 | 권한 없음 |
| 404 | 리소스 없음 |
| 409 | 중복 요청 또는 상태 충돌 |
| 413 | 파일 용량 초과 |
| 500 | 서버 내부 오류 |

## 14. 구현 시 주의사항

본 장은 백엔드 구현 과정에서 반드시 고려해야 하는 주요 검증 사항 및 안정성 관련 주의사항을 정리한다.

상태 기반 처리, 권한 검증, 데이터 무결성 유지, 트랜잭션 처리 등을 중심으로 구현한다.

---

### 14.1 상태 전이 검증

모든 상태 변경 기능은 서버에서 상태 전이 가능 여부를 반드시 검증해야 한다.

클라이언트 요청만으로 상태를 직접 변경하지 않으며,  
정의된 상태 전이 규칙 외의 변경 요청은 차단한다.

---

#### 상태 전이 검증 대상

| 기능 | 검증 내용 |
|---|---|
| MATCH-001 매칭 요청 | 분실물 상태 전이 검증 |
| MATCH-002 매칭 수락 | 매칭/분실물/습득물 상태 전이 검증 |
| MATCH-003 매칭 거절 | 매칭/분실물 상태 전이 검증 |
| MATCH-006 인도 완료 | 모든 관련 상태 전이 검증 |

---

#### 구현 흐름

| 단계 | 설명 |
|---|---|
| 1 | 현재 상태 조회 |
| 2 | 요청 상태 확인 |
| 3 | 상태 전이 가능 여부 검증 |
| 4 | 상태 변경 수행 |

---

### 14.2 중복 매칭 요청 방지

동일 분실물과 동일 습득물에 대해 중복 매칭 요청이 생성되지 않도록 검증해야 한다.

이미 `PENDING` 또는 `ACCEPTED` 상태의 매칭이 존재하는 경우 추가 요청을 차단한다.

분실물 상태가 MATCH_REQUESTED 또는 MATCHING 인 경우 새로운 매칭 요청을 제한한다.

---

#### 구현 시 검증 항목

| 검증 항목 | 설명 |
|---|---|
| 동일 분실물 여부 | 같은 분실물인지 확인 |
| 동일 습득물 여부 | 같은 습득물인지 확인 |
| 기존 매칭 상태 확인 | `PENDING`, `ACCEPTED` 여부 확인 |

---

### 14.3 채팅방 생성 조건 검증

채팅방은 매칭 요청 수락 이후에만 생성 가능하다.

매칭 상태가 `ACCEPTED`가 아닌 경우 채팅방 생성 요청을 허용하지 않는다.

---

#### 구현 시 검증 항목

| 검증 항목 | 설명 |
|---|---|
| 매칭 존재 여부 | 유효한 매칭인지 확인 |
| 매칭 상태 확인 | `ACCEPTED` 상태인지 확인 |
| 기존 채팅방 존재 여부 | 중복 채팅방 생성 방지 |

---

### 14.4 인도 완료 권한 검증

인도 완료 처리는 거래 당사자만 수행할 수 있어야 한다.

매칭과 관련 없는 사용자의 요청은 차단한다.

---

#### 구현 시 검증 절차

| 단계 | 설명 |
|---|---|
| 1 | JWT user_id 조회 |
| 2 | Match 정보 조회 |
| 3 | requester_id / receiver_id 비교 |
| 4 | 거래 당사자 여부 검증 |

---

### 14.5 후기 중복 작성 방지

동일 거래에 대해 동일 사용자가 여러 번 후기를 작성하지 못하도록 검증해야 한다.

후기 작성 전 기존 후기 존재 여부를 확인한다.

동일 사용자 간 반복 거래를 통한 온도 악용 가능성을 고려해야 한다.
추후 동일 사용자 간 반복 후기 제한 정책을 추가할 수 있다.

---

#### 구현 시 검증 항목

| 검증 항목 | 설명 |
|---|---|
| match_id 확인 | 동일 거래 여부 확인 |
| writer_id 확인 | 동일 작성자 여부 확인 |
| 기존 후기 존재 여부 | 중복 후기 차단 |

---

### 14.6 AI 특징 추출 실패 처리

AI 서버 처리 실패 시에도 기본 습득물 등록 기능은 정상적으로 유지해야 한다.

AI 분석 실패로 인해 전체 등록 기능이 실패하지 않도록 구현한다.

---

#### 실패 처리 정책

| 상황 | 처리 방식 |
|---|---|
| AI 서버 연결 실패 | 키워드 없이 등록 유지 |
| AI 응답 시간 초과 | 기본 등록 유지 |
| 키워드 저장 실패 | AI 분석 실패 상태 저장 |

---

#### 권장 처리 흐름

| 단계 | 설명 |
|---|---|
| 1 | 습득물 등록 수행 |
| 2 | AI 분석 요청 |
| 3 | 실패 발생 |
| 4 | analysis_status = FAILED 저장 |
| 5 | 기본 등록 유지 |

---

### 14.7 이미지 업로드 실패 처리

이미지 업로드 실패 시 데이터 무결성을 유지해야 한다.

파일 저장 실패 이후 DB 데이터만 저장되는 상황을 방지해야 한다.

---

#### 실패 처리 정책

| 상황 | 처리 방식 |
|---|---|

### 14.8 DB 트랜잭션 처리 필요 구간
